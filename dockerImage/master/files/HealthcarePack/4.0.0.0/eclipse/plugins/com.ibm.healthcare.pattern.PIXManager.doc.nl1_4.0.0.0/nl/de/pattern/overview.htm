Muster Healthcare: Patient Identifier Cross-reference Manager

Verwenden Sie das Muster Healthcare: Patient Identifier Cross-reference Manager, um einen Patient Identifier Cross-reference Manager zu erstellen, der dem IHE-PIX-Profil (Integrating the Healthcare Enterprise Patient Identifier Cross Referencing) entspricht. Anschließend können Sie Ihre klinischen Anwendungen in das Muster integrieren, die dann als Patient Identity Source- und Patient Identifier Cross-reference Consumer-Akteure gemäß Definition im PIX-Profil agieren.

Lösung

Sie können das Muster Healthcare: Patient Identifier Cross-reference Manager verwenden, um eine Integrationslösung zu erstellen, die ADT HL7-Nachrichten über MLLP von einer klinischen Anwendung, der Patient Identity Source im PIX-Profil, empfängt. Die Integrationslösung gibt dann eine entsprechende Bestätigung an die klinische Anwendung zurück, wie im PIX-Profil angegeben. Wenn die Nachricht gültig ist, verwendet die Integrationslösung SOAP-Anforderung/Antwort-Nachrichten, um den Enterprise Master Patient Index (EMPI) wie durch den Nachrichteninhalt angegeben zu aktualisieren. Der EMPI wird in IBM InfoSphere MDM ausgeführt.

Wenn die ADT-Nachricht vom Typ A01, A04 oder A05 ist, werden die in der Nachricht enthaltenen Patientendetails dem EMPI als neuer Datensatz hinzugefügt. Sie können keinen Patienten hinzufügen, dessen Details mit denen eines bereits vorhandenen Patienten übereinstimmen.

Wenn die ADT-Nachricht vom Typ A08 ist, werden die in der Nachricht enthaltenen Patientendetails verwendet, um einen vorhandenen Datensatz im EMPI zu aktualisieren. Sie können nur einen vorhandenen Patienten mit bereits angegebenen Details aktualisieren.

Alle anderen ADT-Nachrichtentypen führen zu einer AR-Bestätigung (Application Reject). Bei erfolgreichen Aktualisierungen wird, nachdem eine Erfolgsnachricht vom EMPI empfangen wurde, eine AA-Bestätigung (Application Accept) gesendet. Bei nicht erfolgreichen Aktualisierungen wird eine AE-Bestätigung (Application Error) oder eine AR-Bestätigung (Application Reject) gesendet. Weitere Informationen zum Fehler finden Sie in dem Feld "ERR.3.HL7ErrorCode" der Bestätigung, das den HL7-Fehlercode enthält. Die Fehlercodes werden durch HL7 Table 0357 definiert. Wenn dem Fehler ein SOAP-Fehler zugeordnet ist, finden Sie im Feld "ERR.7.DiagnosticInformation" weitere Informationen zum Fehler.

Wenn Sie eine Instanz des Musters Healthcare: Patient Identifier Cross-reference Manager generieren, wird automatisch eine Bibliothek mit dem Namen PIXManagerHL7DFDLLibrary erstellt. Die Bibliothek enthält ein DFDL-Modell der Version 2.5.1, das verwendet wird, um die vom Muster verwendeten HL7-Nachrichten zu definieren.

Bestätigungen

Die folgenden Tabellen beschreiben die Werte in den Bestätigungsnachrichten, die vom Muster zurückgegeben werden.

MSH

Feldname Feldwert in Bestätigungsnachricht Falls in der Eingabenachricht nicht vorhanden
MSH1 - Feldtrennzeichen MSH1 - Feldtrennzeichen (aus der Eingabenachricht) '|'
MSH2 - Servicezeichenfolge MSH2 - Servicezeichenfolge (aus der Eingabenachricht) '^~\&'
MSH3 - Sendende Anwendung Der Wert, der durch den Musterparameter Anwendung in der Musterparametergruppe PIX Manager angegeben wird
MSH4 - Sendende Einrichtung Der Wert, der durch den Musterparameter Einrichtung in der Musterparametergruppe PIX Manager angegeben wird
MSH5 - Empfangende Anwendung Der Wert, der durch den Musterparameter Anwendung in der Musterparametergruppe Patient Identity Source angegeben wird
MSH6 - Empfangende Einrichtung Der Wert, der durch den Musterparameter Einrichtung in der Musterparametergruppe Patient Identity Source angegeben wird
MSH7 - Datum/Uhrzeit der Nachricht Das aktuelle Datum und die aktuelle Uhrzeit im Format YYYYMMddHHmmss .
MSH9 - Nachrichtentyp 'ACK^ACK^ACK'
MSH10 - Nachrichtensteuerungs-ID Neue eindeutige Kennung
MSH11 - Verarbeitende ID MSH11 - Verarbeitende ID (aus der Eingabenachricht) 'P'
MSH12 - Versions-ID MSH12 - Versions-ID (aus der Eingabenachricht) '2.5'

MSA

Feldname Feldwert in Bestätigungsnachricht Falls in der Eingabenachricht nicht vorhanden
MSA.1.AcknowledgmentCode Beim erfolgreichen Empfang der Nachricht wird der MSA-Bestätigungscode auf AA gesetzt. Ansonsten wird der beim erfolgreichen Empfang der Nachricht wird der MSA-Bestätigungscode je nach Fehler auf AR oder AE gesetzt.
MSA.2.MessageControlID MSH10 - Nachrichtensteuerungs-ID (aus der Eingabenachricht) '123456789'

ERR

Feldname Feldwert in Bestätigungsnachricht
ERR.2.ErrorLocation Das HL7-Nachrichtenfeld, das den Fehler enthält.
ERR.3.HL7ErrorCode Der HL7-Fehlercode (Kommunikation).
ERR.4.Severity Der Schweregrad eines Anwendungsfehlers.
ERR.7.DiagnosticInformation Zusätzliche Fehlerinformationen für SOAP-Fehler.
