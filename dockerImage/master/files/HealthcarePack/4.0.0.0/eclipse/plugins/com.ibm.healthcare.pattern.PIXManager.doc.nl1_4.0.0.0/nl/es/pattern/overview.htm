Patrón Servicio sanitario: Patient Identifier Cross-reference Manager

Puede utilizar el patrón Servicio sanitario: Patient Identifier Cross-reference Manager con IBM® InfoSphere® Master Data Management (MDM) para crear un Patient Identifier Cross-reference Manager que cumpla las especificaciones de perfil PIX (Patient Identifier Cross Referencing) IHE (Integrating the Healthcare Enterprise). A continuación puede integrar sus aplicaciones clínicas en el patrón, para actuar como actores de Origen de identidad de paciente y de Consumidor de referencia cruzada de identificador de paciente, según se define en el perfil PIX.

Solución

Puede utilizar el patrón Servicio sanitario: Patient Identifier Cross-reference Manager para crear una solución de integración que reciba los mensajes HL7 de ADT en MLLP de una aplicación clínica (el origen de identidad de paciente del perfil PIX). A continuación, la solución de integración devuelve un acuse de recibo a la aplicación clínica, según se haya especificado en el perfil PIX. Si el mensaje es válido, la solución de integración utiliza mensajes SOAP de solicitud/respuesta para actualizar el EMPI (Enterprise Master Patient Index) según se especifique en el contenido del mensaje. El EMPI se ejecuta en IBM InfoSphere MDM.

Si el mensaje ADT es de tipo A01, A04 o A05, los detalles del paciente que contiene el mensaje se añaden al EMPI como un nuevo registro. No se puede añadir un paciente si ya existe un paciente con esos detalles.

Si el mensaje ADT es de tipo A08, los detalles del paciente que contiene el mensaje se utilizan para actualizar un registro existente en el EMPI. Solo se puede actualizar un paciente si ya existe uno con esos detalles.

Todos los demás tipos de mensaje ADT dan como resultado un acuse de recibo AR (Application Reject). En caso de actualizaciones satisfactorias, se envía un ACK de tipo AA (Application Accept) cuando se recibe una respuesta satisfactoria del EMPI. En caso de actualizaciones no satisfactorias, se envía un ACK de tipo AE (Application Error) o AR (Application Reject). Puede ver información adicional sobre el error en el campo ERR.3.HL7ErrorCode del ACK, que contiene el código de error HL7. Los códigos de error se definen en la Tabla HL7 0357. Si hay un error de SOAP asociado con el error, puede ver más información sobre el error en el campo ERR.7.DiagnosticInformation.

Cuando se genera una instancia del patrón Servicio sanitario: Patient Identifier Cross-reference Manager, se crea automáticamente una biblioteca denominada PIXManagerHL7DFDLLibrary. Esta biblioteca contiene un modelo de DFDL versión 2.5.1, que se utiliza para definir los mensajes HL7 que utiliza el patrón.

Acuses de recibo

En as tablas siguientes se describen los valores de los mensajes de acuse de recibo que devuelve el patrón.

MSH

Nombre del campo Valor del campo en el mensaje de acuse de recibo Si no está presente en el mensaje de entrada
MSH.1.FieldSeparator MSH.1.FieldSeparator del mensaje de entrada '|'
MSH.2.ServiceString MSH.2.ServiceString del mensaje de entrada '^~\&'
MSH.3.SendingApplication El valor que especifica el gestor de Aplicación en el grupo de parámetros de patrón Gestor de PIX
MSH.4.SendingFacility El valor que especifica el gestor de Recurso en el grupo de parámetros de patrón Gestor de PIX
MSH.5.ReceivingApplication El valor que especifica el gestor de Aplicación en el grupo de parámetros de patrón Origen de identidad de paciente
MSH.6.ReceivingFacility El valor que especifica el gestor de Recurso en el grupo de parámetros de patrón Origen de identidad de paciente
MSH.7.DateTimeOfMessage Fecha y hora actuales con el formato AAAAMMddHHmmss
MSH.9.MessageType 'ACK^ACK^ACK'
MSH.10.MessageControlID Nuevo identificador exclusivo
MSH.11.ProcessingID MSH.11.ProcessingID del mensaje de entrada 'P'
MSH.12.VersionID MSH.12.VersionID del mensaje de entrada '2.5'

MSA

Nombre del campo Valor del campo en el mensaje de acuse de recibo Si no está presente en el mensaje de entrada
MSA.1.AcknowledgmentCode Cuando se recibe correctamente el mensaje el código de acuse de recibo MSA se establece en AA. De lo contrario, el mensaje el código de acuse de recibo MSA se establece en AR o en AE, según el error.
MSA.2.MessageControlID MSH.10.MessageControlID del mensaje de entrada '123456789'

ERR

Nombre del campo Valor del campo en el mensaje de acuse de recibo
ERR.2.ErrorLocation El campo del mensaje HL7 que contiene el error
ERR.3.HL7ErrorCode El código de error HL7 (comunicación)
ERR.4.Severity La gravedad de un error de aplicación
ERR.7.DiagnosticInformation Información de error adicional para errores SOAP
