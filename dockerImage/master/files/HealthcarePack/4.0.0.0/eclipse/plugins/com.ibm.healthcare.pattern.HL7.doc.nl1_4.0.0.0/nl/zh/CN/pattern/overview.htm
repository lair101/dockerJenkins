“Healthcare：从 HL7 到 HL7”模式

“Healthcare：从 HL7 到 HL7”模式将能够发送 Health Level Seven International (HL7) V2 消息的应用程序与一个或多个能够接收 HL7 消息的应用程序集成。 这些应用程序必须能够使用基于 TCP/IP 的 Minimal Lower Layer Protocol (MLLP) 发送和接收 HL7 消息。

注：“Healthcare：HL7 到 HL7”模式在 IBM WebSphere Message Broker Connectivity Pack for Healthcare V7.0 中便已可用，而且在本版本的 IBM Integration Bus Healthcare Pack 中仍然可用并受支持。 但是，如有可能，建议您改用“Healthcare：HL7 到 HL7 DFDL”模式。“Healthcare：从 HL7 到 HL7 DFDL”模式使用 DFDL 消息模型而不是 MRM 和 HL7v25P 消息集。DFDL 消息模型具有下列优点：

  • DFDL 是一种开放标准格式，而 MRM 和 HL7v25P 消息集是 IBM Integration Bus 的专有格式
  • DFDL 编辑器提供了用于开发和测试 HL7 模式扩展的工具，与 MRM 和 HL7v25P 消息集相比更为简单
  • DFDL 消息模型支持 HL7 V2.7、V2.6、V2.5.1 及之前版本，而 MRM 和 HL7v25P 消息集仅支持 HL7 V2.5.1 及之前版本

重要事项！

必须先将 HL7 消息集部署到集成节点，然后再部署模式实例。此消息集用于在消息流处理 HL7 消息时对这些消息进行解析。直到部署此消息集之后，此模式生成的消息流才能部署到集成节点。

您可以从模式的资源页面将 HL7 消息集导入到工作空间中。

构建 HL7 消息集需要花费较长时间，并且可能会显著减慢工作空间的速度 - 因此，建议您执行下列操作：

  • 将该 HL7 消息集导入到工作空间中
  • 将该 HL7 消息集添加到 BAR 文件中
  • 将该 BAR 文件部署到集成节点
  • 从工作空间中删除该 HL7 消息集和 BAR 文件

如果要使用图形映射器进行消息变换，那么需要将 HL7 消息集留在工作空间中。通过从消息集中删除 HL7 章节，可以缩短构建该消息集所需的时间。 请将 segments.mxsd、fields.mxsd 和 datatypes.mxsd 消息定义文件保留在消息集中，这些文件是此模式所必需的。

选项

“Healthcare：从 HL7 到 HL7”模式中的“接收方”流支持下列选项：

  • 使用基于 TCP/IP 的 MLLP 接收 HL7 消息
  • 验证并解析消息
  • 审计入局消息
  • 检查并处理重复消息
  • 支持排序选项
  • 将每条入局消息发布或发送到队列，以便在模式外部进行处理
  • 将输出发送到“变换并路由”队列
  • 处理异常

“Healthcare：从 HL7 到 HL7”模式中的 TransformAndRoute 流支持下列选项：

  • 变换为规范 XML 格式
  • 将规范入局消息发布或发送到队列，以便在模式外部进行处理
  • 对 HL7 段中的其他字段进行报告
  • 将消息分发到多个可分别配置的目标
  • 对要发送到各个目标的消息进行过滤
  • 对要针对各个目标包括的段进行过滤
  • 定制点，用于支持包括特定于目标的非标准更改
  • 重新变换为 HL7，并将输出发送到各个目标的“发送方”流的队列
  • 处理异常

“Healthcare：从 HL7 到 HL7”模式中的每个“发送方”流支持下列选项：

  • 为此目标读取变换后的消息
  • 对变换后的消息进行验证
  • 确保根据排序选项按顺序交付消息
  • 在消息未按顺序到达时发出通知
  • 使用基于 TCP/IP 的 MLLP 将 HL7 消息交付到目标应用程序
  • 处理异常
