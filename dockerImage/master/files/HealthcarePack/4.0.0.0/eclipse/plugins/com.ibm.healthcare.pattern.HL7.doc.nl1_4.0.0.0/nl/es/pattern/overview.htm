Patrón Servicio sanitario: HL7 a HL7

El patrón Servicio sanitario: HL7 a HL7 integra una aplicación que puede enviar mensajes HL7 (Health Level Seven International) v2 con una o más aplicaciones que pueden recibir mensajes HL7. Las aplicaciones deben poder enviar y recibir mensajes HL7 utilizando el protocolo MLLP (Minimal Lower Layer Protocol) sobre TCP/IP.

Nota: El patrón Servicio sanitario: HL7 a HL7 está disponible a partir de IBM WebSphere Message Broker Connectivity Pack for Healthcare versión 7.0 y sigue disponible y soportado en esta versión de IBM Integration Bus Healthcare Pack. Sin embargo, se recomienda utilizar el patrón Servicio sanitario: HL7 a DFDL HL7 en su lugar, cuando sea posible. El patrón Servicio sanitario: HL7 a DFDL HL7 utiliza el modelo de mensaje DFDL en lugar de MRM y el conjunto de mensajes HL7v25P. El modelo de mensaje DFDL tiene las siguientes ventajas:

  • DFDL es un formato estándar abierto, mientras que MRM y el conjunto de mensajes HL7v25P son de propiedad de IBM Integration Bus
  • El editor DFDL proporciona herramientas más sencillas para desarrollar y probar extensiones a los esquemas HL7, en comparación con MRM y el conjunto de mensajes HL7v25P
  • El modelo de mensaje DFDL admite HL7 versiones 2.7, 2.6, 2.5.1 y anteriores, mientras que MRM y el conjunto de mensajes HL7v25P solo admiten HL7 versión 2.5.1 y anteriores

Importante

Debe desplegar el conjunto de mensajes HL7 en el nodo de integración antes de desplegar una instancia de patrón. El conjunto de mensajes se utiliza para analizar mensajes HL7 a medida que los procesan los flujos de mensajes. Los flujos de mensajes que genera el patrón no se pueden desplegar en el nodo de integración hasta que se haya desplegado el conjunto de mensajes.

Puede importar el conjunto de mensajes HL7 en el espacio de trabajo desde la página Recursos para el patrón.

El conjunto de mensajes HL7 tarda mucho tiempo en construirse y puede ralentizar significativamente el espacio de trabajo. Por esto se recomienda hacer lo siguiente:

  • Importe el conjunto de mensajes HL7 en su espacio de trabajo
  • Añada el conjunto de mensajes HL7 a un archivo BAR
  • Despliegue el archivo BAR en su nodo de integración
  • Suprima el conjunto de mensajes HL7 y el archivo BAR de su espacio de trabajo

Si desea utilizar el correlacionador de gráficos para las transformaciones de mensajes, tendrá que dejar el conjunto de mensajes HL7 en su espacio de trabajo. Puede reducir el tiempo que se tarda en construir el conjunto de mensajes suprimiendo los capítulos HL7 del conjunto de mensajes Deje los archivos de definición de mensajes segments.mxsd , fields.mxsd y datatypes.mxsd del conjunto de mensajes tal como los requiera el patrón.

Opciones

El flujo Receiver del patrón Servicio sanitario: HL7 a HL7 admite les siguientes opciones:

  • Recepción de mensajes HL7 utilizando MLLP sobre TCP/IP
  • Validación y análisis de los mensajes
  • Auditoría de los mensajes entrantes
  • Comprobación y proceso de mensajes duplicados
  • Soporte a las opciones de secuenciación
  • Publicación o envío a una cola de cada uno de los mensajes entrantes para su proceso fuera del patrón
  • Envío de la salida a una cola de transformación y direccionamiento
  • Manejo de excepciones

El flujo TransformAndRoute del patrón Servicio sanitario: HL7 a HL7 admite les siguientes opciones:

  • Transformación a formato XML canónico
  • Publicación o envío a una cola del mensaje canónico entrante para su proceso fuera del patrón
  • Informes de los campos adicionales encontrados en los segmentos HL7
  • Distribución de los mensajes a diversos destinos, configurables por separado
  • Filtrado de los mensajes a enviar a cada destino
  • Filtrado de los segmentos a incluir para cada destino
  • Puntos de personalización para habilitar la inclusión de cambios no estándar específicos de cada destino
  • Volver a transformar a HL7 y envío de la salida a la cola del flujo Sender de cada destino
  • Manejo de excepciones

Cada uno de los flujos Sender del patrón Servicio sanitario: HL7 a HL7 admite les siguientes opciones:

  • Lectura de mensajes transformados para este destino
  • Validación de los mensajes transformados
  • Garantizar que los mensajes se entregan secuencialmente según las opciones de secuenciación
  • Notificar cuando los mensajes llegan fuera de secuencia
  • Entrega de los mensajes HL7 a la aplicación de destino utilizando MLLP sobre TCP/IP
  • Manejo de excepciones
