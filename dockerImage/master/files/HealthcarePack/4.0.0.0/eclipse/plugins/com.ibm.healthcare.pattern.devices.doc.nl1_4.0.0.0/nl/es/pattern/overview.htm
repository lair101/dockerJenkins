Patrón Servicio sanitario: Dispositivos médicos a EMR

El patrón Servicio sanitario: Dispositivos médicos a EMR integra dispositivos médicos con una aplicación de Historial clínico electrónico (HCE) que pueda recibir mensajes de resultados de observación (ORU R01) de HL7 (Health Level Seven International) v2. La aplicación debe poder recibir mensajes HL7 utilizando el protocolo MLLP (Minimal Lower Layer Protocol) sobre TCP/IP.

Nota: Este patrón está disponible a partir de IBM WebSphere Message Broker Connectivity Pack for Healthcare versión 7.0 y sigue disponible y soportado en esta versión de IBM Integration Bus Healthcare Pack. No obstante, este patrón todavía utiliza MRM y el conjunto de mensajes HL7v25P en lugar del modelo de mensaje DFDL que se introdujo en IBM WebSphere Message Broker Connectivity Pack for Healthcare versión 8. Se recomienda el modelo de mensaje DFDL para aplicaciones nuevas y actualizadas porque tiene las siguientes ventajas:

  • DFDL es un formato estándar abierto, mientras que MRM y el conjunto de mensajes HL7v25P son de propiedad de IBM Integration Bus
  • El editor DFDL proporciona herramientas más sencillas para desarrollar y probar extensiones a los esquemas HL7, en comparación con MRM y el conjunto de mensajes HL7v25P
  • El modelo de mensaje DFDL admite HL7 versiones 2.7, 2.6, 2.5.1 y anteriores, mientras que MRM y el conjunto de mensajes HL7v25P solo admiten HL7 versión 2.5.1 y anteriores

Importante

Debe desplegar el conjunto de mensajes HL7 en el nodo de integración antes de desplegar una instancia de patrón. El conjunto de mensajes se utiliza para analizar mensajes HL7 a medida que los procesan los flujos de mensajes. Los flujos de mensajes que genera el patrón no se pueden desplegar en el nodo de integración hasta que se haya desplegado el conjunto de mensajes.

Puede importar el conjunto de mensajes HL7 en el espacio de trabajo desde la página Recursos para el patrón.

El conjunto de mensajes HL7 tarda mucho tiempo en construirse y puede ralentizar significativamente el espacio de trabajo. Por esto se recomienda hacer lo siguiente:

  • Importe el conjunto de mensajes HL7 en su espacio de trabajo
  • Añada el conjunto de mensajes HL7 a un archivo BAR
  • Despliegue el archivo BAR en su nodo de integración
  • Suprima el conjunto de mensajes HL7 y el archivo BAR de su espacio de trabajo

Si desea utilizar el correlacionador de gráficos para las transformaciones de mensajes, tendrá que dejar el conjunto de mensajes HL7 en su espacio de trabajo. Puede reducir el tiempo que se tarda en construir el conjunto de mensajes suprimiendo los capítulos HL7 del conjunto de mensajes Deje los archivos de definición de mensajes segments.mxsd , fields.mxsd y datatypes.mxsd del conjunto de mensajes tal como los requiera el patrón.

Si utiliza búsquedas de base de datos para la identificación de pacientes y configura la instancia de patrón para crear un servicio web, debe importar y desplegar el conjunto de mensajes PatientIdentifiersService. Los nodos SOAP del flujo de servicios web requieren este conjunto de mensajes porque contiene el archivo WSDL. Este conjunto de mensajes y también un conjunto de mensajes que modela las medidas del nodo MedicalDeviceInput están disponibles en la página Recursos para el patrón.

El conjunto de mensajes del nodo MedicalDeviceInput se puede utilizar con el correlacionador gráfico para transformar los mensajes de los dispositivos. Asegúrese de definir DataObject como analizador de destino para el mensaje de salida en el correlacionador gráfico, porque el valor por defecto es XMLNSC. Este patrón espera que todos los mensajes de dispositivos estén en el dominio DataObject y no funcionará correctamente si utiliza XMLNSC.

Opciones

El flujo Dispositivos médicos de este patrón admite las siguientes opciones:

  • Recepción de mensajes de dispositivos médicos. Se crea un servicio configurable predeterminado para generar datos a partir de un controlador de dispositivos virtual.
  • Publicación o envío a una cola de cada uno de los mensajes entrantes para su proceso fuera del patrón
  • Envío de la salida a la cola de Transformación y direccionamiento
  • Establecimiento de un tiempo de caducidad, de modo que los mensajes de dispositivo antiguos se descarten

El flujo Transformación y direccionamiento de este patrón admite las siguientes opciones:

  • Transformación a un formato HL7 canónico
  • Publicación o envío a una cola del mensaje canónico para su proceso fuera del patrón
  • Puntos de personalización para habilitar la inclusión de cambios no estándar específicos de cada destino
  • Transformación a HL7 y envío de la salida a la cola del flujo Sender
  • Asociación de los mensajes de dispositivos a identificadores de pacientes mediante una búsqueda en una base de datos, utilizando una base de datos definida como parte de este patrón
  • Se incluyen instrucciones para crear las tablas de base de datos para DB2 u Oracle y se llega a la base de datos mediante ODBC desde el patrón

El flujo Sender de este patrón admite las siguientes opciones:

  • Lectura de mensajes transformados para el destino
  • Validación de los mensajes transformados
  • Entrega de los mensajes HL7 a la aplicación de destino utilizando MLLP sobre TCP/IP

Opcionalmente, este patrón puede generar un servicio web para gestionar la base de datos de identificadores de pacientes:

  • Servicio web implementado como un flujo de mensajes para obtener, definir y borrar identificadores de pacientes
  • Registrar todas las solicitudes de servicio web en una cola
  • Implementa el servicio web como consultas o actualizaciones a una base de datos

Todos los flujos de mensajes que genera este patrón manejan las excepciones utilizando colas para guardar la información de error.
