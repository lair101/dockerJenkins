Muster 'Healthcare: Medizinische Geräte an EKA'

Das Muster 'Healthcare: Medizinische Geräte an EKA' werden medizinische Geräte in eine EKA-Anwendung (Elektronische Krankenakte) integriert, die Untersuchungsergebnisnachrichten (ORU R01) des Standards HL7 v2 (Health Level Seven International) empfangen kann. Die Anwendung muss in der Lage sein, HL7-ORU-R01-Nachrichten unter Verwendung des MLLP-Protokolls (MLLP = Minimal Lower Layer Protocol) über TCP/IP zu empfangen.

Hinweis: Dieses Muster steht in IBM WebSphere Message Broker Connectivity Pack for Healthcare Version 7.0 zur Verfügung. Es ist auch in der vorliegenden Version von IBM Integration Bus Healthcare Pack verfügbar und wird dort noch unterstützt. Allerdings verwendet es nach wie vor noch MRM und die HL7v25P-Nachrichtengruppe anstatt des DFDL-Nachrichtenmodels, das in IBM WebSphere Message Broker Connectivity Pack for Healthcare Version 8 eingeführt wurde. Das DFDL-Nachrichtenmodell wird für neue und aktualisierte Anwendungen empfohlen, da es folgende Vorteile bietet:

  • DFDL ist ein Format, das auf offenen Standards basiert, während MRM und die HL7v25P-Nachrichtengruppe zu IBM Integration Bus gehören.
  • Im Vergleich zu MRM und der HL7v25P-Nachrichtengruppe stellt der DFDL-Editor einfachere Tools für die Entwicklung und den Test von Erweiterungen für das HL7-Schema bereit.
  • Das DFDL-Nachrichtenmodell unterstützt HL7 Version 2.7, 2.6, 2.5.1 und ältere Versionen, während MRM und die HL7v25P-Nachrichtengruppe nur HL7 Version 2.5.1 und ältere Versionen unterstützen.

Wichtig:

Sie müssen die HL7-Nachrichtengruppe vor der Implementierung einer Musterinstanz im Integrationsknoten implementieren. Mit der Nachrichtengruppe werden HL7-Nachrichten bei ihrer Verarbeitung durch die Nachrichtenflüsse syntaktisch analysiert. Die vom Muster generierten Nachrichtenflüsse können erst nach der Implementierung der Nachrichtengruppe im Integrationsknoten implementiert werden.

Sie können die HL7-Nachrichtengruppe aus der Seite Ressourcen für das Muster in Ihren Arbeitsbereich importieren.

Da die Erstellung der HL7-Nachrichtengruppe lange dauert und Ihren Arbeitsbereich erheblich verlangsamen kann, wird die folgende Vorgehensweise empfohlen:

  • Importieren Sie die HL7-Nachrichtengruppe in Ihren Arbeitsbereich.
  • Fügen Sie die HL7-Nachrichtengruppe einer BAR-Datei hinzu.
  • Implementieren Sie die BAR-Datei im Integrationsknoten.
  • Löschen Sie die HL7-Nachrichtengruppe und die BAR-Datei aus Ihrem Arbeitsbereich.

Wenn Sie den grafischen Mapper für Nachrichtentransformationen verwenden möchten, müssen Sie die HL7-Nachrichtengruppe in Ihrem Arbeitsbereich behalten. Sie können die Erstellungszeit für die Nachrichtengruppe verkürzen, indem Sie die HL7-Kapitel aus der Nachrichtengruppe löschen. Die Nachrichtendefinitionsdateien segments.mxsd , fields.mxsd und datatypes.mxsd müssen in der Nachrichtengruppe verbleiben, da sie vom Muster benötigt werden.

Wenn Sie eine Datenbanksuche für die Identifikation von Patienten verwenden und die Musterinstanz für die Erstellung eines Web-Service konfigurieren, müssen Sie die Nachrichtengruppe 'PatientIdentifiersService' importieren und implementieren. Diese Nachrichtengruppe wird von den SOAP-Knoten im Nachrichtenfluss 'Web Service' benötigt, da sie die WSDL-Datei enthält. Diese Nachrichtengruppe sowie eine Nachrichtengruppe, die die Messungen vom MedicalDeviceInput-Knoten modelliert, stehen auf der Seite Ressourcen für das Muster zur Verfügung.

Die Nachrichtengruppe des MedicalDeviceInput-Knotens kann in Verbindung mit dem grafischen Mapper für Nachrichtentransformationen von Gerätenachrichten verwendet werden. Achten Sie darauf, dass Sie den Zielparser auf 'DataObject' für die Ausgabenachricht im grafischen Mapper setzen, da er standardmäßig den Wert XMLNSC annimmt. Da dieses Muster alle Gerätenachrichten in der DataObject-Domäne erwartet, funktioniert es nicht ordnungsgemäß, wenn Sie XMLNSC verwenden.

Optionen

Der Nachrichtenfluss 'Medical Devices' in diesem Muster unterstützt die folgenden Optionen:

  • Empfangen von Nachrichten aus medizinischen Geräten - ein standardmäßiger konfigurierbarer Service wird für die Generierung von Daten aus einem Treiber für virtuelle Einheiten erstellt.
  • Veröffentlichen oder Senden an eine Warteschlange (jede eingehende Nachricht für die Verarbeitung außerhalb des Musters)
  • Senden der Ausgabe an die Warteschlange 'Transform and Route'
  • Festlegen einer Ablaufzeit, damit veraltete Gerätenachrichten gelöscht werden

Der Nachrichtenfluss 'Transform and Route' in diesem Muster unterstützt die folgenden Optionen:

  • Transformation in ein kanonisches HL7-Format
  • Veröffentlichen oder Senden an eine Warteschlange (die kanonische Nachricht für die Verarbeitung außerhalb des Musters)
  • Anpassungspunkte, damit zielspezifische Änderungen einbezogen werden können
  • Transformation in HL7 und Senden der Ausgabe an die Warteschlange für den Nachrichtenfluss 'Sender'
  • Zuordnen von Gerätenachrichten zu Patienten-IDs über eine Datenbanksuche unter Verwendung einer Datenbank, die als Teil dieses Musters definiert ist
  • Anweisungen für die Erstellung der Datenbanktabellen für DB2 oder Oracle stehen zur Verfügung; die Datenbank ist über ODBC vom Muster aus erreichbar.

Der Nachrichtenfluss 'Sender' in diesem Muster unterstützt die folgenden Optionen:

  • Lesen transformierter Nachrichten für das Ziel
  • Validierung transformierter Nachrichten
  • Zustellung von HL7-Nachrichten an die Zielanwendung unter Verwendung von MLLP über TCP/IP

Dieses Muster kann optional einen Web-Service generieren, mit dem die Datenbank der Patienten-IDs verwaltet wird:

  • Implementierung des Web-Service als Nachrichtenfluss, damit Patienten-IDs abgerufen, festgelegt und gelöscht werden können
  • Protokollierung aller Web-Service-Anforderungen, die an eine Warteschlange gestellt werden
  • Implementierung des Web-Service als Datenbankabfragen oder -aktualisierungen

Alle von diesem Muster generierten Nachrichtenflüsse verarbeiten Ausnahmebedingungen mithilfe von Warteschlangen, in denen die Fehlerinformationen gespeichert werden.
