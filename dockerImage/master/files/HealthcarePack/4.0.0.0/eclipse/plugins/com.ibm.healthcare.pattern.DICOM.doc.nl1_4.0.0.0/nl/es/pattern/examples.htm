Mensajes de ejemplo para solicitudes C-FIND y C-MOVE

Los mensajes de consulta del patrón Servicio sanitario: Servicio web a DICOM especifican sus criterios de búsqueda utilizando etiquetas DICOM. El estándar DICOM define los nombres de etiqueta, su significado y los valores permisibles. En cada raíz y nivel de consulta, se define una etiqueta como única (U). Un valor único en una etiqueta única define de forma exclusiva una única entidad a ese nivel. Es decir, dos entidades en el mismo nivel no tienen el mismo valor de etiqueta única. Los SCP C-FIND admiten la existencia y coincidencia de todas las etiquetas únicas definidas por DICOM. Todas las entidades gestionadas por SCP C-FIND deben tener un valor de etiqueta único de longitud distinta a cero.

Cada raíz y nivel de consulta DICOM tiene un pequeño conjunto de etiquetas obligatorias (Required: R) que todo proveedor DICOM debe admitir. Las etiquetas obligatorias suponen que el SCP de un C-FIND debe admitir la coincidencia en base a un valor contenido en un solicitud C-FIND. Varias entidades pueden tener el mismo valor para las etiquetas obligatorias. Es decir, un valor diferenciado en una etiqueta obligatoria no identifica necesariamente a una entidad individual en el nivel de la clave. Los SCP C-FIND admiten la existencia y coincidencia de todas las etiquetas obligatorias definidas a una raíz y nivel dado. Si un SCP C-FIND gestiona una entidad con una etiqueta de longitud cero, el valor se considera desconocido y cualquier coincidencia contra la etiqueta de longitud cero se considera una coincidencia satisfactoria.

Hay un conjunto mucho más grande de etiquetas opcionales (O) que los proveedores DICOM pueden admitir. Los proveedores son libres de añadir etiquetas adicionales admitidas en los criterios de búsqueda. Las etiquetas opcionales contenidas en una solicitud C-FIND pueden tener tres tipos distintos de comportamiento según el soporte de existencia y/o coincidencia del SCP C-FIND. En primer lugar, si el SCP C-FIND no admite la existencia de una etiqueta opcional, la etiqueta no se devuelve en una respuesta C-FIND. En segundo lugar, si el SCP admite la existencia de una etiqueta opcional, pero no admite coincidencias en la etiqueta, esta se procesa de la misma manera que una etiqueta obligatoria de longitud cero. Es decir, el valor especificado para ser coincidentes con la etiqueta se ignora, pero el SCP puede devolver un valor en el mensaje de respuesta. Finalmente, si el SCP da soporte a la existencia y la coincidencia de la etiqueta, esta se procesa de la misma manera que una etiqueta obligatoria.

DICOM tiene muchas opciones para describir criterios de búsqueda C-FIND, concretamente respecto a comodines, rangos de fechas y horas y coincidencia de varios valores en una lista. Consulte el PS3.4 del estándar DICOM para obtener más información. El nodo DICOMFindMove actualmente no da soporte a la coincidencia en las secuencias. Un mensaje de solicitud XML en el nodo DICOMFindMove puede solicitar etiquetas de devolución adicionales. Estos códigos no se utilizan para los criterios de búsqueda en la solicitud C-FIND al SCP. Si el SCP admite la etiqueta devuelta, los mensajes de respuesta contienen un valor coincidente para cada etiqueta de búsqueda y un valor para cada etiqueta de devolución.

En el ejemplo siguiente se muestra un mensaje de consulta con dos etiquetas de búsqueda de 00100010 (PatientName) y 00080050 (AccessionNumber), y una etiqueta de devolución adicional configurada de 00080054 (RetrieveAETitle):

<DICOM:FindMove xmlns:DICOM="http://com.ibm.healthcare/DICOM">
    <QueryRoot>STUDY</QueryRoot>
    <QueryLevel>STUDY</QueryLevel>
    <Match>
        <Attribute Tag="00100010">FEROVIX</Attribute>
        <Attribute Tag="00080050">1210490</Attribute>
    </Match>
    <Return>
        <Attribute Tag="00080054"/>
    </Return>
</DICOM:FindMove>

El nombre de etiqueta normalmente es más fácil de entender y menos propenso a errores que utilizar la etiqueta. Como se muestra a continuación, los mensajes de respuesta siempre utilizan la etiqueta DICOM.

Mandatos C-MOVE

Los mensajes de solicitud C-MOVE se configuran exactamente igual que los mensajes de solicitud C-FIND. La única diferencia es la inclusión de un elemento MoveDestination en el mensaje de consulta XML. Este destino del movimiento es el título de la entidad de aplicación donde se deben trasladar las instancias DICOM utilizando mandatos C-STORE. El PACS, que procesa los mandatos C-FIND y C-MOVE debe estar previamente configurado con los detalles de conexión del destino del movimiento.

En el ejemplo siguiente se muestra un mensaje DICOM de solicitud C-FIND y C-MOVE con MoveDestination especificado:

<DICOM:FindMove xmlns:DICOM="http://com.ibm.healthcare/DICOM">
    <QueryRoot>PATIENT</QueryRoot>
    <QueryLevel>STUDY</QueryLevel>
    <MoveDestination>REMOTEAE</MoveDestination>
    <OperationPriority>NORMAL</OperationPriority> 
    <Match>
        <Attribute Tag="PatientID">fWT0KTL</Attribute>
        <Attribute Tag="AccessionNumber">1210490</Attribute>
    </Match>
    <Return>
    </Return>
</DICOM:FindMove>

El campo OperationPriority es un campo opcional del mensaje de solicitud XML. Depende del proveedor si se utiliza o no. El campo puede tener los valores HIGH, NORMAL o LOW.

Los mensajes de respuesta SOAP del patrón se devuelven una vez que se ha completado el mandato C-MOVE. Por lo tanto, la respuesta puede tardar algún tiempo si se trasladan ficheros grandes de estudios, series o imágenes. Los mensajes de respuesta también incluye un elemento XML denominado MoveResults que describe el éxito general, o no, del mandato C-MOVE. A continuación se muestran los campos de estado detallados del mensaje de respuesta (Completed, Failed y Warnings).

        <Attribute Tag="00080056" VR="CS">ONLINE</Attribute>
        <Attribute Tag="0020000D" VR="UI">1.3.12.2.1107.5.1.4.54023.30000004093013443132800000021</Attribute>
        <Attribute Tag="0020000E" VR="UI">1.3.12.2.1107.5.1.4.54023.30000004093016410718700003676</Attribute>
        <Attribute Tag="00200013" VR="IS">188</Attribute>

Raíz de estudio

La raíz de estudio es similar a la raíz de paciente y esta similitud a menudo provoca confusiones. No hay consulta a nivel de paciente en la raíz de estudio. El primer nivel admitido es la consulta a nivel de estudio. La consulta a nivel de estudio tiene un mayor conjunto de atributos permitidos que se pueden utilizar. Los criterios de búsqueda admitidos incluyen la fecha del estudio, la hora del estudio y también atributos de paciente tales como el nombre del paciente, el ID de paciente y el sexo del paciente. Por lo tanto, esta consulta a nivel de estudio se puede considerar una fusión de las consultas a nivel de estudio y de paciente de la raíz de paciente.

La consulta a nivel de estudio resulta muy útil porque se pueden consultar de una sola vez todos los estudios de un paciente dados unos criterios de búsqueda básicos como nombre del paciente y fecha del estudio. La raíz de paciente separa esto en dos consultas distintas.

Las consultas a nivel de serie y a nivel de imagen son idénticas en la raíz de estudio y en la raíz de paciente.
